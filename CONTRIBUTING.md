# Contributing / Development Notes

## ⚠️ Critical: Node Version for Local Development

**Before running `npm install` to regenerate `package-lock.json`, ALWAYS switch to the same Node version that the CI runner uses:**

```bash
nvm use 24
npm install
```

The GitHub Actions workflow uses `NODE_VERSION: 24` (which comes with npm 11). If you generate `package-lock.json` with an older Node/npm version (e.g., Node 22 / npm 10), `npm ci` on the runner will fail with:

```
npm error `npm ci` can only install packages when your package.json and package-lock.json are in sync.
```

This happens because npm 10 and npm 11 produce structurally different lockfiles, even though both claim `lockfileVersion: 3`.

**Rule: Always match the local npm version to the CI npm version when touching package-lock.json.**

## Lessons Learned (from initial setup)

These issues were encountered and solved during development. They're documented here so they don't repeat:

| # | Issue | Root Cause | Fix |
|---|-------|-----------|-----|
| 1 | `Rollup failed to resolve @aws-amplify/data-schema-types` | Package missing from dependencies | Add `@aws-amplify/data-schema-types` to `dependencies` (NOT devDependencies) |
| 2 | `ERR_MODULE_NOT_FOUND: Cannot find package 'tsx'` | `@aws-amplify/backend-cli` needs tsx at runtime | Add `tsx` to `devDependencies` |
| 3 | `TS2688: Cannot find type definition file` | TypeScript auto-discovers broken empty `@types/` dirs | Add explicit `"types": ["react", "react-dom"]` to tsconfig.json |
| 4 | `npm ci` fails with lockfile version mismatch | Lockfile generated with npm 10, runner has npm 11 | Use `npm install` in CI (or regenerate lockfile with matching npm version) |
| 5 | Vite tries to bundle backend packages | `@aws-amplify/backend` was in `dependencies` | Move to `devDependencies` — it's build-time only |
| 6 | `amplify_outputs.json` missing breaks `tsc` | File is generated by sandbox, not committed | Keep an empty `{}` stub locally (gitignored) |
| 7 | `BootstrapNotDetectedError` in CI | CDK not bootstrapped in target region | Run `npx cdk bootstrap aws://ACCOUNT/REGION` once |
| 8 | GitHub Actions workflow fails on first push | OIDC + IAM role must exist before workflow runs | Run `bootstrap-github-actions.sh` BEFORE pushing workflow file |
| 9 | `gh auth login --with-token` requires `read:org` | gh CLI validates all standard scopes | Use `export GH_TOKEN=...` instead (no scope validation) |
| 10 | `AmplifyFullAccess` policy doesn't exist | Not a real AWS managed policy | Use inline policy with specific amplify:* actions instead |

## Dependency Placement Rules

```
dependencies (bundled into frontend):
  - aws-amplify               → Frontend SDK
  - @aws-amplify/data-schema-types → Required peer dep at runtime
  - react, react-dom          → UI framework

devDependencies (build/deploy-time only, NOT bundled):
  - @aws-amplify/backend      → Schema definition (CDK, not browser)
  - @aws-amplify/backend-cli  → ampx CLI for deployments
  - tsx                       → Required by backend-cli internally
  - typescript, vite, vitest  → Build tools
```

**Never put `@aws-amplify/backend` in `dependencies`** — Vite will try to bundle it and fail because it imports Node-only CDK packages.

## CI Workflow Notes

- The workflow uses `npm install` (not `npm ci`) for cross-version compatibility
- node_modules are cached by `package-lock.json` hash — so `npm install` only runs on dependency changes
- Amplify backend artifacts (`.amplify/`) are cached by `amplify/` directory hash
- The `cleanup` job auto-deletes Amplify branches when Git branches are deleted

## Bootstrap Script

The `scripts/bootstrap-github-actions.sh` script handles all one-time AWS + GitHub setup. It:
- Auto-installs `gh` CLI if missing (supports dnf, apt, direct download)
- Prompts interactively for GitHub token (or uses `GH_TOKEN` env var)
- Creates OIDC provider, IAM role, CDK bootstrap, Amplify app
- Scopes IAM policies to the configured region only
- Sets GitHub repo variables automatically

Run it from **AWS CloudShell** (recommended) or any environment with AWS CLI access.
